Overview

ABC Microfinance Uganda Ltd is a Tier 3 Microfinance Deposit-taking Institution (MDI) licensed and regulated by the Bank of Uganda under the Microfinance Deposit-taking Institutions Act. The organisation serves over 12,000 registered clients across three offices in Kampala, Jinja, and Mbarara with a staff of 34.

This project designed and implemented a centralised relational database system in MySQL — normalised to Third Normal Form — to replace a fragmented Excel-based records system. A PHP web application provides full CRUD operations for all staff roles without requiring SQL knowledge.

MySQL 8.0 PHP / Apache 3NF Normalization Role-Based Access Control CIA Triad Security Bank of Uganda Compliant

Problem

The organisation managed all client records, loan portfolios, and transaction histories using Microsoft Excel spreadsheets and paper forms — with no shared database or network connection between branches. Staff transferred data manually via USB drives, WhatsApp messages, and email attachments.

Data Duplication
The same client registered differently in each branch — inconsistent and conflicting records with no way to reconcile them
Data Loss Risk
Financial records stored on individual USB drives — permanently lost if a device fails, is stolen, or is damaged
Slow Loan Processing
Approving a loan required physically transferring documents between offices — delays of several days for urgent requests
Zero Access Control
Any staff member could view or modify any record — violating confidentiality requirements and Bank of Uganda regulations

Conceptual Design — Entity-Relationship Diagram

Six entities were identified from the requirements analysis: Branch, Staff, Member, Account, Loan, and Transaction_Log. All relationships use one-to-many cardinality. The Staff table includes a recursive self-referencing foreign key (ManagerID) implementing the management hierarchy. The Loan table has two separate foreign keys to Staff — ApprovedBy and DisbursedBy — enforcing segregation of duties.

ER diagram showing all six entities, primary keys, foreign key relationships, and cardinality notation

Logical Design — Database Schema

The schema was implemented in MySQL with tables created in parent-before-child order to satisfy foreign key constraints. Each table includes a primary key with AUTO_INCREMENT, NOT NULL constraints on mandatory fields, UNIQUE constraints to prevent duplicates, and CHECK constraints to enforce business rules.

Normalization — Excel to 3NF

The starting point was the organisation's flat Excel spreadsheet where member details, branch information, account balances, and loan data were all combined in a single sheet. The schema was normalised through three normal forms.

1NF
Eliminated repeating groups
Kamya John appeared in multiple rows because he had two accounts — his name, phone, and NationalID were duplicated in every row. Each attribute now contains one atomic value, stored once.
2NF
Separated mixed facts
A single row contained facts about four different things: the member, the branch, the account, and the loan. Each table now describes exactly one entity — all non-key attributes depend fully on the primary key.
3NF
Removed transitive dependency
BranchCity was determined by BranchName, not MemberID — a transitive dependency (MemberID → BranchName → BranchCity). BranchName and BranchCity moved to the Branch table. Now if a branch changes city, the update happens in exactly one place.

System Architecture

The application follows a three-tier client-server architecture. All three offices connect to the same application server and database through a secure WAN connection — ensuring every branch always works with the same live data, eliminating the USB drive data transfer problem entirely.

Three-tier architecture — Kampala, Jinja, and Mbarara branch browsers connect via secure WAN to Apache/PHP application server, which connects to MySQL at Kampala HQ

Application Interface — CRUD Operations

The PHP web application provides six main screens covering all CRUD operations. The interface was designed for staff with varying computer literacy — plain language labels, large input fields, no SQL knowledge required. A persistent sidebar provides role-filtered menu access based on the logged-in user's role.

Dashboard (READ) — branch managers see real-time KPI cards: total members, accounts, savings (UGX 5,995,000), active loans (4), outstanding loans (UGX 2,720,000), and monthly trend charts
CREATE Add New Member
Tellers register new clients with full name, gender, phone, National ID, and branch assignment. Three save options: Save, Save & Continue (for group registrations), Save & Close.
READ Member Financial Summary
Searchable, filterable member directory showing each member's total savings and outstanding loan balance — a JOIN query across Member, Account, and Loan tables.
CREATE Loan Application
Loan officers submit applications and manage the loan portfolio. The table below shows real loan data including amount, repaid, outstanding balance, and status (Active/Pending) with Approve action buttons.
UPDATE Account Management
Staff search by Account ID, the system retrieves the member name, current account type, and balance. Branch managers can update the account type through the dropdown — restricted by role-based access control.

Advanced SQL — Views and Queries

Four views were created to allow staff to query complex data without SQL knowledge. The BranchPerformance_View consolidates member counts, total savings, and outstanding loan balances per branch using a multi-table LEFT JOIN with CASE WHEN aggregate functions.

BranchPerformance_View in MySQL Workbench — a multi-table LEFT JOIN with COUNT DISTINCT, SUM, and CASE WHEN aggregates producing a consolidated branch dashboard in a single query

Security — Role-Based Access Control

Following the principle of least privilege, four database user roles were created with GRANT statements assigning only the permissions each role requires. The GRANT statements were executed in MySQL Workbench and are visible in the screenshot below.

GRANT statements in MySQL Workbench — Teller, Loan Officer, Branch Manager, and IT Administrator roles each receive only the permissions necessary for their job function

🔐 Segregation of duties: the Loan Officer who approves a loan (ApprovedBy FK) must be a different person from the Teller who disburses it (DisbursedBy FK) — enforced at both schema level and application level.

Outcome

Data Integrity
3NF eliminates duplication — one source of truth for all 12,000+ client records accessible from all three offices simultaneously
Processing Speed
Loan approvals no longer require physical document transfer — branch managers approve from the dashboard, tellers disburse immediately
Security
Role-based GRANT statements replace open Excel files — each staff member accesses only what their role requires, Bank of Uganda compliant
Audit Compliance
Soft deletion with IsActive flag, DeletedAt timestamp, and DeletedBy FK retains all records for the 7-year regulatory retention requirement
